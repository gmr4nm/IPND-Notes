Lesson 4.1 - Introduction to Networks


Basic Definitions

  • Network - a group of entities that can communicate, even though they are not all directly connected. A group of entities that are all directly connected is technically not a network.

  • Latency - the time that it takes for a message to get from the source to intended recipient of the message.

  • Bandwidth - amount of information that can be transmitted per unit of time.

  • Bit - a bit is the smallest unit of information. In a computational binary system, a bit is either a 1 or a 0. Each time you add a bit in a binary system, you theoretically double the amount of information you can transmit.

  • Protocol - a set or rules that determine how two entities talk to each other.

  • Flooding the network - A message from one node in a network spreads to each available node at each step of propagation, resulting in every node receiving the message at least once. This is highly inefficient.

Basic Principles

The two main ways the measure a network are latency and bandwidth. Latency is generally a property of the geographical distance and the number and quality of nodes that data is passing through. Bandwidtch can be related to most network variables other than distance.

To make a functional network, you need ways to encode and interpret messages, specific ways to route messages, and rules for deciding who gets to use resources. The HyperText Transfer Protocol (HTTP) fulfils this need for most browser-server communications on the internet.


Lesson 4.2 - Make the Internet Work for You


URL - Uniform Resource Locator

A URL is is a functional command that specifies locations and communicates with a server in the typical web browswer environment. A URL typically specifies a protocol, a host, defines a path, and may contain other information such as queries, fragments, or specify which port(s) are used for specific operations.

A Diagramed Example:

http://example.com:8000/somepath/foo?parameter=bar#afragment

  1. http → Protocol
  2. example.com → Host
  3. :8000 → Port 8000
  4. /somepath/foo → Path
  5. ?parameter=bar → Query
  6. #afragment → Fragment

Relative vs. Absolute Paths

In the above example, the host is not diagramed as part of the path. This is only true if you are talking about relative paths (relative to the host or the page where the link appears). The absolute path will also contain the host information and the protocol.

Using query parameters

In the example:

http://example.com/path?p=1&g=foo&h=bar

Note that the first parameter is separated by a ? and following queries are separated by &’s. A large number of parameters can be defined for a server in this way.

Using Fragments

A fragment is separated from the rest of the URL by a hashtag. It generally follows the parameters. It is not sent to the server, but it is used to reference a part of the page that you are looking at and exists purely in the browser. Note that queries are part of the path, but fragments never are.

Using Ports

In order to make a connection with a machine, you need an address of the machine (host) and you need a port. The default port is 80. If you want to specify a port, you place that in between the host and the path.

HTTP Requests (browser to server)

There are three components to an HTTP request line: the method, the path, and the version.
For example:

http://www.example.com/foo/apic.png?p=1#somefrag

might look something like this:

GET (method) /foo/apic.png?p=1 (path) HTTP/1.1 (version)
Host: www.example.com (host line header)
Accept: image/gif, image/jpeg, */*
Accept-Language: en-us
User-Agent: Mozilla 4.0 (user agent header)

HTTP Responses (server to browser)

An example of the reply format:

HTTP/1.1(version) 200(status code) OK(Reason phrase)
Server: Apache /2.2.3 (note: you generally don’t want to give this information to would-be hackers…)
Content-Type: test/html;
Content-Length: 1539

Static vs. Dynamic Responses

  • Static responses: use pre-written files or images to respond to requests.

  • Dynamic responses: make up the generated content on the fly and respond with it. A web application is what generates this content.

GET vs POST

  • GET - used for fetching documents, has a maximum URL length, is OK to cache, and shouldn’t change the server

  • POST - used for updating data, has no max length, is not OK to cache, and is OK to change the server

GET requests should NEVER change the server. There are all sorts of browser plug-ins that will automatically send GET requests for any number of reasons. Allowing these to change the server is DANGEROUS and does not meet industry standards.


Lesson 4.3 - Forms


Forms - A Functional Class of HTML

Forms are an HTML class (generally displayed as an input box) that allow users to input data on a webpage. The form class is defined by the <form> </form> tags. Usually opening tag will define an attribute, 'action,' like so:

<form action="http://someurl.com">

The action attribute defines a URL where the form input is sent.

Form Methods

<form method="get|post">

GET appends form data into the URL(name/value pairs in the query) and should NEVER be used to send sensitive data. As previously mentioned, it should never be used to change data on the server. This method can be very useful for defining specific content that a user might wish to bookmark.

POST does not append data into the URL and is therefore suitable or handling sensitive data. There are also no inherent size limits associated with POST data.

Labels

Labels organize and define content inside of a form class. These are much cleaner than just entering raw text and allow you to define both input and labels as a specific class in your web framework or CSS.

Input

The HTML input class is defined by the <input> </input> tags. These tags are defined within a form class. Inputs are usually defined by type, name, and value (although there are many other attributes they can have).

Other Input Options

The <select> </select> tags are used to define dropdown selection menus (where <option> </option> tags are used to specify options). Name and value attributes can be specified to specify how query strings are appended.


Lesson 4.4 - Python Break-Modulus and Dictionaries


Break-Modulus

The python break-modulus can be useful for defining relationships between number sets. It defines r where r is related to a and b such that:

a % b = r

where a / b = q with remainder r

Dictionaries

Unlike lists, dictionaries are mutable. This is because, unlike a list, a dictionary has a key associated with each value. Also, dictonaries have no inherent order (why would they need it if you can look any value up with a key?).

A dictionary is defined:

some_dictionary = {key1:value1, key2:value2}

Values can be queried/modified:

dictionary[key] = some_value

It is possible to have a dictionary within a dictionary. Querying or modifying a value within a dictionary is as simple as:

some_dictionary["Parent_key"][“Daughter_key”] = daughter_key_value


Lesson 4.5 - Validation


Validation

Validation is the process of verifying on the server side that we received the expected input. Commonly, this would be checking to make sure input doesn't contain certain values defined by Regex expressions or verifying that input isn't 'empty.' It might also include content filters such as the profanity filter used in my guestbook page.

Escaping

Escaping allows you to prevent text that is input into your website from being read as HTML or JS code. This is commonly accomplished be replacing characters like '>' with equivalent HTML expressions (&gt;). Doing this prevents low-level malicious attacks on your website and also allows your user to input a wider range of characters should they feel the need.

String Substitution

The string:

<bold tag> Some bold text! </bold tag>

Can be defined dynamically in one place, and then substituted into the correct placement between the HTML tags like so:

a = “some bold text”

<bold tag> %s </bold tag> %a

Additionally, multiple strings can be substituted like so:

<bold tag> %s %s %s… </bold tag> %(a, b, c…)

The number of strings must correspond to the number of substitution commands and substitutions occur in the order they are defined in parenthesis. You can also use a dictionary function to make it easier to keep track of strings, or because you want to reuse the same strings multiple times in different places:

<bold tag> %(key1)s %(key2)s %(key1)s… </bold tag> %{“key1”: some_value, “key2”: some_value}


Lesson 4.6 - HTML Templates


Template Inheritance

What if you don't want to have to write and compile repetitive header/footer content for each page of your website? HTML templates can help!

Using the property of template inheritance, it is simple to write one parent template that includes (for example, the header/footer) information and have any number of daughter webpages inherit this information and generate their own individual content around the code that they have inherited from the parent template.

Web frameworks make this process much simpler and scalable. They allow a developer to avoid unnecessary repitition in his or her code by facilitating the creation of HTML templates that will propogate code (as desired) anywhere on a website. For this course, I've primarily worked within the web framework Jinja2.

Inheritance in Jinja2

In order to save time and effort with template inheritance, you will need to create a parent template with the relevant header/footer code. Then add:

{% block content %}
{% endblock %}

where you want your dynamically generated content to be displayed in the base template(parent.html). In your daughter templates, put {% extends “parent.html” %} at the top of the template to automatically inherit content from parent.html.

Then you put:
{% block content %}

your_dynamically_generated_content_here

{% endblock %}

This expression will match your block content from the daughter page with the header/footer/styling info from your base.html.

Jinja2 Syntax

{{some_variable}}

{{}} - really means print and references the variable inside the curly braces.

Statement Syntax:

{% some_statement %}

output

{% end_statement %}

A for loop would be defined like so:

{% for_statement %}

body

{% endfor %}


Lesson 4.7 - Databases


What is a Database?

A database is a program that stores and retrieves data (usually large amounts of structured data). Usually databases are communicated with via the server that is actually running your website. Browser to database communication is less common.

A database is generally made up of tables (columns v. rows). Usually the first column is related to identity so that each row has an ID (usually an integer) associated with it.

Querying

There are a number of downsides to querying data in Python: the process is error-prone, tedious, and computationally slow. This is the reason databases are necessary and special. The most commonly used query languages are variants of SQL (Structured Query Language). Fortunately, it isn't too difficult to import SQL into Python and wrap SQL requests inside Python functions.

A query function has the generic syntax:

SELECT * FROM links WHERE id = some_id;

A common example of this might look like:

“select * from links where id = 5 order by votes asc;”

Joins

This type of SQL query uses data from multiple tables to query both. Essentially, this is useful where the query from one table is used to determine the scope of another query. Example where links, user are the two tables:

“select link* from links, user where link.user_id = user.id and user.name = SPEZ;”

Indexes

Hash tables in Python would be accessed using sequential scans. This would be way too slow with millions of links…especially if the goal is to create a responsive website. Indexes are the solution to this.

ACID Principles

Use ACID principles to make your database as functional as possible. Note that there will always be some tradeoffs between these different principles. You can’t completely do all of these.

  1. Atomicity - All parts of a transaction succeed or fail together.
  2. Consistency - The database will always be consistent.
  3. Isolation - No transaction can interfere with another.
  4. Durability - Once a transaction is accomplished, it will not be lost.
